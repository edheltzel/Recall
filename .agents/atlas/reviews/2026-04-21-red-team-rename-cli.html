Blocker Hooks SessionRecall.ts and SessionPreCompact.ts read MEM_DB_PATH directly — missing from scope Both ›

Both hook files carry their own getDbPath() implementation that reads the env var directly — they don't import from src/. They're copied verbatim to ~/.claude/hooks/ at install time.

Evidence
hooks/SessionRecall.ts:46 — if (process.env.MEM_DB_PATH) return process.env.MEM_DB_PATH;
hooks/SessionPreCompact.ts:74 — identical pattern
Failure mode
After rename + v1.1.0 removes the fallback, the CLI still works but SessionRecall / SessionPreCompact silently break — because the hooks were never updated.
Fix in plan
New Commit 2 dedicated to hook updates. Each hook reads RECALL_DB_PATH first, MEM_DB_PATH as fallback, same chain as src/db/connection.ts.

High severity

Real issues with non-trivial blast radius. None are runtime-fatal, but each causes downstream pain.

High Worktree workflow leaves dangling symlinks after ExitWorktree Both ›

CLAUDE.md mandates worktrees, but this task is a real exception. bun link from inside .claude/worktrees/rename-mem-to-recall/ symlinks ~/.bun/bin/recall to the worktree's dist/index.js. When the worktree is removed, the symlink dangles and recall becomes a broken command.

Correct flow
Commits inside worktree → merge to main → ExitWorktree action: remove → run ./install.sh from main.
Fix in plan
Explicit "Execution order" section now mandates merge-then-install from main, with recall_verify_global_link() catching regressions.
High Live MCP server dies when tsup --clean wipes dist/ — not at restart Contrarian ›

The plan notes "restart Claude Code after ./install.sh" but underestimates when breakage starts. bun run build (implicit in install) uses tsup --clean, which deletes dist/ before rebuild. The live MCP server process holds the old file open via inode, but the new build creates a new inode. During the install → restart window, memory_* tools return connection errors.

Fix in plan
CHANGELOG migration note rewritten: "Finish any in-progress memory_dump before running ./install.sh, then restart Claude Code."
High install.sh:90-98 hardcodes ~/.bun/bin/mem as mem_bin General ›

Step 5 of the installer looks for the binary at the old path. After recall_cleanup_legacy removes it and bun link creates recall instead, the "$mem_bin" init call fails with "binary not found."

Dependency
Tight: the cleanup (removes old symlink) and the variable rename (updates the lookup) must land in the same commit. They do — both in Commit 4.
High Deprecation console.warn would pollute every test run General ›

If the fallback emits a console.warn when MEM_DB_PATH is the source, setupTestDb() triggers the warning on every test that uses the harness — noisy, and some CI runners fail on unexpected stderr.

Design correction
Plan now uses silent fallback (no warning). Combined with moving the harness update to Commit 1, fallback is never exercised in tests.

Design corrections

Where the contrarian reviewer changed the shape of the plan — not by finding a bug, but by proposing a cleaner approach.

Corrected Drop src/lib/env.ts — getDbPath() already exists Contrarian›

The prior plan proposed a new shared helper file. But getDbPath() is already a named, exported function in src/db/connection.ts:15-17 — with exactly one caller. A new module would add indirection for zero reuse.

Before (plan v1)
New file: src/lib/env.ts
export function getDbPath()
Callers: 1
After (plan v2)
Modify in place: connection.ts:15-17
No new file
Callers unchanged
Corrected Kill the console.warn "once per process" guard — no call site justifies it Contrarian›

The guard protects against multiple warnings per run, but no command calls getDbPath() more than once. The guard is complexity for nothing. Combined with General's observation that the warning would pollute tests, the cleanest answer is: silent fallback for v1.0.x, hard removal in v1.1.0.

Corrected Version bump → 1.0.0, not 0.8.0 Contrarian›

Semver for a single-user CLI is purely communicative. The v2 foundation (decisions#49) already shipped the architectural work 1.0.0 would represent. The rename is the name catching up to the architecture — this IS 1.0.0.

Corrected Update test harness in Commit 1, not Commit 5 Contrarian›

Moving setup.ts env-var rename to Commit 1 means the fallback is never exercised by the test suite → zero deprecation noise, zero stderr pollution, cleanest path through all intermediate commits.

Decisions the plan held

Red team challenged — plan stood its ground for the right reasons.

Upheld MCP tool names (memory_search, etc.) stay unchanged Contrarian›

Tool names are the stable public API. The memory_ prefix is semantically accurate — these tools operate on memory regardless of project name. Agents call tools by bare name, so there's no renaming pressure from the CLI change.

Upheld MCP server ID (recall-memory) stays unchanged Contrarian›

Original rationale "breaks every agent prompt" overstated the cost, but the decision stands: cosmetic rename, zero user value. "Recall is the project, memory is what it stores" is still a coherent mental model.

Upheld Atomic rename, not phased Contrarian›

Two-phase (ship both names, then remove old) would need two releases, two changelogs, and bin coexistence complexity. For a single user with 6 tests (after correction), atomic is simpler and the fallback mitigates the test transition.

Upheld recall_cleanup_legacy placement (before recall_link_global) Contrarian›

Cleanup first → link fresh → configure MCP. Both functions that touch settings.json are idempotent; running both is safe. The apparent coupling with recall_configure_mcp is benign: cleanup fixes stale path, configure skips (correctly) because the ID is already registered.
