# Suite C post-dedup run manifest (issue #78)

Reproducible record of the `recall dedup --execute` → re-run Suite C → diff-vs-baseline
measurement that closes the **Hardening & Dedup Proof** milestone.

## What was measured

The committed 2026-06-11 Suite C baseline recorded search() precision under noise with
**dedup not run**. This run re-records the same seeded corpora **after** applying
`recall dedup --execute`, so the diff is dedup's efficacy report — the empirical evidence
the parked entity-keying gate (#49) is waiting on.

## Exact invocation

```bash
# From repo root, on the worktree branch (deps #63 + #70 already on main).
RECALL_BENCH_C_DEDUP=1 bun run benchmarks/runner.ts C
```

`RECALL_BENCH_C_DEDUP=1` enables an opt-in dedup pass added to `runSuiteC` for this issue
(default OFF — the no-dedup baseline path is unchanged). For each corpus size, between
`seedFixture()` and the measured queries, the harness runs the **real** dedup code path,
mirroring `recall dedup --execute` exactly:

```ts
const plan = planDedup(db);                              // tables: all; threshold: 0.95 (default); semantic: on
applyDedupPlan(db, plan, { destructive: false });        // --execute → MARK (no --delete); records hidden from search via dedup_lineage
```

- **Non-destructive.** `recall dedup --execute` *marks* duplicates; it does not delete them
  (that needs `--delete`). `search()` excludes records marked in `dedup_lineage`.
- **Semantic pass skipped.** `planDedup` uses **stored** embeddings only. The seeded corpus
  has none (Suite C never embeds; `recall dedup --execute` does not backfill), so only the
  **exact** pass runs — identical to what any user without `recall embed backfill` would get.

## Integrity checks (the run is real, not an all-zeros artifact)

1. **Silent-zero canary (#83) passed** — smallest corpus (100) produced a nonzero aggregate
   relevance score, so `runSuiteC` returned rather than throwing.
2. **`getLastSearchErrors()` clean** — `runQuery` aborts the whole run if `search()` swallows
   any per-table SQL error; the run completed, so no errors were swallowed.
3. **Dedup demonstrably exercised** — `dedup_marked` = 0 / 5 / 288 / 8,328 across the ladder
   (8,621 total). Recorded as `dedup_planned` / `dedup_marked` samples in the result JSONL.
4. **Reproduction / inertness proof** — re-running Suite C with dedup **OFF** on this machine
   reproduced the committed 2026-06-11 baseline **byte-for-byte across all 76 relevance
   metrics**. This proves the harness change is inert when off, the baseline is deterministic,
   and every post-dedup delta is attributable solely to dedup.

## Result (headline)

Exact-only dedup moves MRR negligibly; the precision-under-noise collapse is intact. See
`*.diff.md` for the full table. This is the documented empirical case for #49 (lightweight
entity keying): exact + stored-embedding dedup is structurally unable to touch the
near-duplicate / entity-collision noise that buries true targets, because that noise is
*near*, not *exact*, and there are no stored embeddings for the semantic pass.
