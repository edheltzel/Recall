# Suite C embedding-backed dedup run manifest (issue #99)

Reproducible record of the **embedding backfill → `recall dedup --execute` (semantic pass
active) → re-run Suite C → diff** measurement that answers the #49 decision input: does an
embedding-backed semantic dedup pass at threshold 0.95 lift Suite C precision-under-noise?

## What was measured

The #78 capstone proved shipped-default `recall dedup --execute` moves Suite C MRR negligibly —
but its semantic pass was a **no-op** because the corpus had no stored embeddings. This run
populates embeddings first (`recall embed backfill` semantics), so the semantic pass actually
runs at cosine ≥ 0.95, then re-records the same seeded corpora. The diff vs the 2026-06-11
baseline AND vs #78's exact-only result is the embedding path's efficacy report.

**Headline result:** the embedding-backed semantic pass does **not** lift precision — at 10k it
*lowers* it (MRR 0.1095 → 0.0179) by auto-merging 5 ground-truth targets into near-duplicate
noise. Full analysis in `2026-06-18T17-14-50-suite-C.diff.md`.

## Exact invocation

```bash
# From repo root, on the worktree branch. Requires a live embedding service.
RECALL_BENCH_C_DEDUP=1 \
RECALL_BENCH_C_EMBED_BACKFILL=1 \
RECALL_BENCH_C_SIZES=100,1000,10000 \
bun run benchmarks/runner.ts C
```

`RECALL_BENCH_C_EMBED_BACKFILL=1` enables the opt-in embedding-backed variant added for this
issue (default OFF — both the no-dedup baseline path and the #78 exact-only dedup path are
unchanged; proven below). It requires `RECALL_BENCH_C_DEDUP=1` and a reachable embedding
service, and **throws rather than fabricating vectors** if the service is unavailable. For each
corpus size, between `seedFixture()` and the measured queries, the harness runs:

```ts
// 1. Backfill embeddings for EVERY dedup-eligible record (all 5 DEDUP_TABLES),
//    using the production primitives: embeddingTextFor (same per-table content
//    composition recall embed backfill uses) + embed (real Ollama client) +
//    embeddingToBlob, written to the same `embeddings` table the CLI writes.
const backfilled = await backfillCorpusEmbeddings(db);
// 2. Real dedup code path, identical to `recall dedup --execute`:
const plan = planDedup(db);                       // tables: all; threshold: 0.95 (DEFAULT_SEMANTIC_THRESHOLD); semantic: on
applyDedupPlan(db, plan, { destructive: false }); // MARK (no --delete); records hidden from search via dedup_lineage
```

- **Threshold 0.95** is `DEFAULT_SEMANTIC_THRESHOLD` — the shipped CLI default, the value the
  issue specifies. No threshold was invented or overridden.
- **Backfill coverage is a superset of the CLI.** The harness embeds all five `DEDUP_TABLES`
  (messages, decisions, learnings, breadcrumbs, loa_entries), including breadcrumbs, which the
  `recall embed` CLI omits. This gives the semantic pass maximal coverage — the most favorable
  possible conditions for the embedding hypothesis. It still did not help.
- **Non-destructive.** Duplicates are *marked*, not deleted; `search()` excludes records marked
  in `dedup_lineage`.

## Integrity proofs (re-run at HEAD, sizes 100/1k/10k, 5 repeats)

1. **Inertness on the #78 path** — `dedup` ON + `embedBackfill` **OFF** reproduced #78's
   exact-only result byte-for-byte across all 57 relevance metrics (0 mismatches).
2. **Baseline path unchanged** — `dedup` **OFF** reproduced the 2026-06-11 baseline byte-for-byte
   across all 57 relevance metrics (0 mismatches).
3. **Semantic pass demonstrably exercised** — `dedup_semantic_pairs` = 0 / 53 / 2,647 and
   `dedup_marked` = 0 / 58 / 2,935 across the ladder (recorded as samples in the result JSONL).
   `dedup_semantic_pairs` counts the semantic pairs that survived greedy strongest-first +
   sticky-survivor filtering and were planned/actioned (one marked duplicate each), not every
   raw pair `findSemanticPairs` returned at cosine ≥ 0.95.
4. **False-merge mechanism verified** — direct inspection of the 10k dedup plan found 5
   ground-truth targets marked as semantic duplicates of near-dup survivors (cosine 0.976–0.993).

## Result (headline)

The embedding-backed semantic pass at 0.95 does **not** lift Suite C precision-under-noise: inert
at 100/1k, **net-negative at 10k** (MRR 0.1095 → 0.0179), infeasible at 100k. It is the clearest
evidence yet **for** #49's design constraints (passive facets, **no** automatic entity merge):
auto-merging near-duplicates by embedding similarity removes the very targets search needs.
Entity-keying (passive, non-merging) — not an embedding auto-merge pass — remains the right
direction. See `*.diff.md`.
